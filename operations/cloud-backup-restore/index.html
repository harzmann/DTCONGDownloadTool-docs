    Cloud backup and restore¶

    1. Scope and responsibility¶

    FLEET Mira backs up immutable tachograph archive objects and consistent SQLite snapshots to one active Google Drive or HTTPS WebDAV target. The local archive remains primary. Backup is one-way; remote changes are never synchronised back automatically.

    Financial originals use the same active provider infrastructure but a separate financial-documents/... namespace and retention rules.

    2. Remote layout¶

    DLTNG/<installation-id>/
  archives/YYYY/MM/<archive-id>.ddd.gz[.enc]
  database/YYYY/MM/archive_YYYY-MM-DD_HHmmssZ.db.gz[.enc]
  manifests/...
  financial-documents/YYYY/<invoice-id>/<document-id>_<sha256>.<ext>[.enc]

    DLTNG is the compatibility storage prefix and is not renamed to FLEET Mira. Renaming it would orphan existing objects.

    3. Status model¶

    Status Meaning
    Not configured no active target
    Pending durable upload job exists
    Uploading provider transfer in progress
    Verified remote object has expected size/hash
    Failed retryable/permanent error recorded
    Missing remotely previously verified object no longer exists
    Retained after local deletion remote DDD retained by policy

    The archive list shows “backup present” only for Verified.

    4. Target setup¶

    Open Administration → Backups:

    1. choose Google Drive or WebDAV;
    2. authenticate/select target folder;
    3. keep client-side encryption enabled unless policy explicitly permits plaintext;
    4. upload/read back a random test object;
    5. export and safely store the recovery key;
    6. confirm the recovery-key procedure;
    7. activate target and start initial database backup;
    8. monitor initial archive queue until verified.

    WebDAV¶

    Provide HTTPS URL, username, app password and root folder. HTTP is rejected. The test performs OPTIONS, PROPFIND, collection creation, upload and read-back. Upload prefers temporary object + MOVE; if unsupported, direct upload is not published as verified until full read-back succeeds.

    Google Drive¶

    Use the published FLEET Mira desktop OAuth client or import an approved organisation client definition. Scope is limited to drive.file. My Drive and Shared Drives are supported. Refresh tokens are DPAPI-protected. Uploads are resumable.

    5. Encryption¶

    Client-side encryption uses AES-256-GCM with authenticated metadata. The target key is DPAPI-protected locally. The exported recovery-key file is the disaster-recovery dependency and must be stored offline/separately.

    Never store the recovery key only in the same cloud target. Losing both Windows DPAPI context and recovery key makes encrypted backups unrecoverable.

    6. Archive backup flow¶

    flowchart TD
    A[Local archive transaction] --> B[Archive row + backup job committed]
    B --> C[Compute/confirm compressed SHA-256]
    C --> D[Optional AES-GCM encryption]
    D --> E[Upload temporary/resumable object]
    E --> F[Remote stat/read-back]
    F --> G{Size and hash match?}
    G -->|Yes| H[Verified status + audit]
    G -->|No| I[Failed/conflict + retry]

    Retries occur after 1, 5, 15 and 60 minutes and then no more often than every six hours. A process restart resumes durable pending jobs. Same archive ID with different remote content is a conflict and is never overwritten silently.

    7. Strict terminal-deletion mode¶

    Normally local archiving completes the download and cloud backup follows asynchronously. A task may enable Wait for cloud backup before deleting from terminal. In strict mode, mark_processed and deletion are deferred until Verified. A later run recognises the local duplicate, continues its pending backup and deletes the terminal object only after verification.

    8. Database snapshots¶

    Default schedule: daily 02:00 UTC, with same-day catch-up after application restart.

    1. create a consistent file using SQLite Backup API;
    2. run PRAGMA integrity_check;
    3. hash SHA-256;
    4. GZIP compress;
    5. optionally encrypt;
    6. upload and verify remotely.

    Retention keeps the latest 30 daily snapshots and one monthly snapshot for 12 months. Only expired database snapshots are automatically removed; DDD archive objects are never automatically deleted.

    9. Restore one archive object¶

    1. Select the archive row and choose Restore from backup.
    2. Confirm administrative permission and reason where required.
    3. Download to a temporary file.
    4. Verify remote size/hash, decrypt and verify authenticated data.
    5. Verify compressed and decompressed hashes.
    6. Atomically move to the expected local archive path.
    7. Reconcile status; do not automatically re-activate a logically deleted business row.

    Failure leaves the current local object untouched and removes temporary partial data.

    10. Database restore¶

    flowchart TD
    A[Select remote snapshot] --> B[Download to temporary location]
    B --> C[Hash/decrypt/decompress]
    C --> D[SQLite integrity_check]
    D --> E[Stop schedulers and background writers]
    E --> F[Create safety copy of current DB]
    F --> G[Atomically replace database]
    G --> H[Restart application and run migrations]
    H --> I[Reconcile archive/backup state]

    Database replacement is staged for application restart when the live connection cannot be safely closed. If preflight validation fails, no replacement request is committed.

    11. Full disaster recovery¶

    1. Install the same or compatible FLEET Mira version on an isolated host/path.
    2. Configure provider access without starting normal imports.
    3. Supply the recovery key if encrypted.
    4. restore and validate the selected database snapshot;
    5. start once, apply compatible migrations and stop background writes;
    6. enumerate manifests and restore all missing archive/document objects;
    7. verify compressed/decompressed archive hashes and document SHA-256;
    8. reconcile database links, backup status, retention and audit;
    9. test representative C/M analysis, user sign-in and finance originals;
    10. document approval before production cutover.

    12. Target change¶

    A target cannot be deleted while documents reference it. Use the target-change workflow to copy every retained object to the new target, read back/verify it and update references only after success. Keep the old target until a complete reconciliation report is approved.

    13. Permissions and audit¶

    • backup.read: view status;
    • backup.run: retry/verify jobs;
    • backup.manage: configure target/encryption;
    • backup.restore: restore data.

    Target changes, tests, key export confirmation, durable failures, snapshots, retention deletion and restore are audited. Secrets, OAuth tokens and keys are never included.

    14. Operational checklist¶

    • daily: pending/failed counts and last DB snapshot;
    • weekly: target quota/authentication and random archive verification;
    • monthly: database retention and document-vault completeness;
    • quarterly: isolated database + C/M + financial-document restore;
    • after credential/target/version change: connection test and immediate snapshot;
    • incident: preserve logs/audit and do not overwrite the last known-good database.
